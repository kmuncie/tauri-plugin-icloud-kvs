# Design: `examples/demo-app`

The example/demo application for `tauri-plugin-icloud-kvs`: a key-value
editor plus (eventually) a live change-event log, doubling as the plugin's
manual test rig.

## Status

| Date       | Status   | Author                      |
|------------|----------|-----------------------------|
| 2026-07-13 | Accepted | Kevin Muncie (with Claude)  |

## Purpose

- Fulfil the M1.5 "example app" deliverable early so it can serve as the
  host app for the M1.3 iOS simulator checkpoint and the M1.4 change-event
  demo (decision 2026-07-13: build it once instead of a throwaway scratch
  app).
- Act as the permanent manual verification rig described in
  `DEVELOPERS.md` (entitled, signed builds exercising the real KVS store,
  cross-device sync checks).
- Serve as readable example code for third-party Tauri developers
  evaluating the plugin.

## Decisions (settled during brainstorming)

- **Frontend: vanilla TypeScript + Vite.** No framework — neutral,
  minimal example code any plugin evaluator can read. One HTML page, one
  `main.ts`, one CSS file.
- **Scope now: KV editor + placeholder event-log pane.** The event-log
  pane is laid out but shows a "external-change events land in M1.4"
  note; M1.4 replaces the placeholder with a live subscribed list.
- **Plugin consumption: path dependencies.** Rust
  `tauri-plugin-icloud-kvs = { path = "../.." }`, npm
  `"tauri-plugin-icloud-kvs-api": "file:../../guest-js"`. Nothing is
  published yet, and an in-repo example should always test current
  source. (The definition-of-done requirement to consume the *published*
  plugin applies to Team Times, not this example.)
- **CI: light checks only.** `cargo check` on the app's Rust crate plus
  `npm ci && npm run build` on its frontend — catches plugin API drift
  without full `tauri build` cost. The app is not a workspace member of
  the plugin crate.
- **Signing identity is never committed.** Entitlements file ships in the
  repo; team/identity selection is documented in the app's README.

## Layout

```text
examples/demo-app/
├── index.html            # single page, two panes
├── package.json          # vite + typescript + tauri CLI + plugin API (file:)
├── tsconfig.json
├── vite.config.ts
├── src/
│   ├── main.ts           # all UI logic
│   └── styles.css
├── src-tauri/
│   ├── Cargo.toml        # tauri app + plugin path dependency
│   ├── tauri.conf.json   # bundle id, macOS entitlements wiring
│   ├── entitlements.plist
│   ├── capabilities/default.json   # icloud-kvs:default + core defaults
│   ├── build.rs
│   └── src/
│       ├── main.rs       # stock Tauri main
│       └── lib.rs        # builder + .plugin(tauri_plugin_icloud_kvs::init())
└── README.md             # run instructions, signing/entitlement setup
```

## UI (single window, two panes)

- **Header:** account-status badge showing `available` / `noAccount`
  (fetched on load and on window focus), and a **Synchronize** button
  that displays the returned boolean.
- **KV editor pane:**
  - Table of every key/value from `getAll()`; values pretty-printed JSON.
  - Per-row **Delete** button; **Refresh** button re-fetches.
  - Add/edit form: key text input + value textarea. The value is parsed
    as JSON; if parsing fails, it is treated as a plain string
    (convenience for quick manual tests). Clicking a row loads it into
    the form for editing.
  - Plugin errors (invalid key, value too large, serialization) render
    in an inline error bar — surfacing real error messages is part of
    the test-rig role.
- **Event-log pane:** pane exists with placeholder copy ("external-change
  events land in M1.4"). M1.4 swaps in a live event list under the same
  layout.

## Entitlements & signing

- `src-tauri/entitlements.plist` sets
  `com.apple.developer.ubiquity-kvstore-identifier` to
  `$(TeamIdentifierPrefix)$(CFBundleIdentifier)`; `tauri.conf.json`
  references it via the macOS bundle `entitlements` option.
- iOS receives the same capability through the Xcode project generated by
  `tauri ios init` (documented in the app README; the generated project
  is not committed).
- Unsigned/unentitled dev builds still run but the store is inert (see
  `DEVELOPERS.md`); the README states this prominently so nobody debugs
  a "broken" demo.

## CI

Add a `demo-app` job to `.github/workflows/ci.yml` (macOS runner):
`cargo check` in `examples/demo-app/src-tauri` and
`npm ci && npm run build` in `examples/demo-app`. No `tauri build`, no
signing in CI.

## Testing

The app **is** the manual test rig; it has no automated UI tests. CI
coverage is compile-level only (above). `DEVELOPERS.md`'s manual
verification protocols (macOS cross-device sync, iOS simulator) are
updated to use this app instead of "a scratch app".

## Out of scope

- Change-event subscription UI (M1.4).
- Publishing artifacts of the demo app; it is never distributed.
- Windows/Linux behavior beyond the plugin's `UnsupportedPlatform` error
  showing in the error bar.

## Alternatives considered

- **Vue 3 frontend** — matches the author's daily stack and standards,
  but adds framework weight to neutral example code. Rejected.
- **Throwaway scratch app for the M1.3 checkpoint** — fastest to the
  checkpoint but discarded afterward; the demo app is an M1.5
  deliverable anyway. Rejected in favor of building once.
- **Full `tauri build` in CI** — highest confidence but several minutes
  per run and signing complications. Rejected for light checks.
- **Published-package dependencies** — impossible pre-publish and wrong
  for an in-repo example that must track source. Rejected.
